The Agent Stack.

A market analysis of how AI-native software shops are building with coding agents in 2026 — and where one engineer's home-grown skills package fits on the map.

Filed   May 16, 2026 · Updated May 17 Subject   STX Skills · v1.9.0 Verdict   8 → 9 / 10

The shops that built the first generation of "AI software engineers" — Cognition, Cosine, Factory, Sourcegraph, All Hands — have converged on a stack. Plan in one place. Edit in worktrees. Verify with tests. Gate at the boundaries that matter. The interesting question isn't whether agents can write code in 2026. They can. The question is whether the orchestration around them is auditable, parallel, and disciplined enough to trust. This report walks the landscape, names the patterns, and scores one specific setup against the public bar.

§ I

The 8090 Question.

8090 — Chamath Palihapitiya's AI-native SDLC company, founded January 2025 and publicly launched September 1 of that year — is the name most often invoked when engineers ask "who's actually doing this well?" Its flagship product is the Software Factory, a four-module pipeline that produces structured upstream context for coding agents: Refinery turns business intent into requirements; Foundry resolves architecture decisions; Planner generates executable work orders with a code-search sub-agent; Validator converts production feedback back into new tasks. A knowledge graph stitches the four together, enriching context as it flows downstream.

The crucial detail: 8090 doesn't run coders itself. The Software Factory is agent-agnostic — it hands off to Devin, Claude, or whatever model is in vogue. Their bet is that context quality matters more than coder identity. In March 2026 they co-launched EY.ai PDLC with Ernst & Young on top of this pipeline.

Their bet is that the upstream context — not the coder — is the moat. — On 8090's architectural wager
§ II

The Landscape.

The rest of the field sorts into a small number of architectural bets. Two stand out: single-orchestrator-with-subagents (Cognition, Cosine, Sourcegraph) versus true multi-agent coordination (Factory, All Hands). One outlier — Poolside — is betting the whole shop on a single RL-trained agent and skipping the orchestration question entirely.

§ III

What "Advanced" Looks Like.

Anthropic's 2026 Agentic Coding Trends Report names multi-agent coordination as the headline shift of the year, citing 3–5× gains on complex tasks. Strip the marketing and the converging stack is concrete. Six patterns show up almost everywhere:

01

The non-editing coordinator

A top-level orchestrator that plans, dispatches, and reviews — but is forbidden from touching source files itself. Factory makes this explicit; Devin enforces it via phase.

02

Role-specialized workers

Coder, reviewer, sometimes a dedicated QA. Specialization by role is far more common than specialization by tier (db / api / ui).

03

Worktree isolation per agent

Concurrent worktrees are the practical containment mechanism. Teams report four to eight per developer as the working ceiling before review overhead breaks parity.

04

Tests as the trust mechanism

Test-driven verification anchors the loop. Factory, Devin and Cosine all gate "task done" on a green test, not the coder's self-assessment.

05

Gates at the boundary, not the step

Humans approve at plan acceptance and at PR merge. Between those points, agents run with delegated autonomy. Step-by-step approval has lost.

06

AGENTS.md + skills + MCP

The portable config standard. Persona files, named skills, and a model-context-protocol surface for tools — the same three primitives across every shop.

§ V

What's Missing.

Four gaps separated this setup from the 9–10 tier when the report first filed. Two are now closed. Each gap below carries its current status — what shipped, what's next, what's still open.

01

Personas live inline.

Resolved in v1.8.0 (Wave 1, May 17 2026). Nine personas now live under .claude/agents/*.md with YAML frontmatter and independent semver. SKILL.md loads each persona by reference; the installer copies the directory into every consuming project; AGENTS.md at the repo root names the portable standard. Every wave records its persona_versions snapshot in wave-state.json. Persona reusability moved 4.5 → 9.

delivered in half a day · PR #8 · merged a8a9f24's parent
02

No automated Reviewer droid.

Resolved in v1.9.0 (Wave 2, May 17 2026). The Reviewer persona sits between Dev hand-back and QA's rerun. Reads the Dev's diff line-by-line, applies a structured checklist (scope, pattern, result-shape, typing, smallest-change, comment-debt, error-handling, smell, out-of-scope-frozen), and emits a verdict. Three instant-halt verdicts catch the test-bypass attempts that previously slipped through to QA: test-file-edit-detected, assertion-weakened, sut-mocked. Verdicts append to reviewer_verdicts[]. Automated review moved 2.5 → 8.

delivered in one day · PR #9 · merged a8a9f24
03

No execution-feedback loop.

Suspicious-change rate, iteration counts, Reviewer rejection rate per tier, most-common concern kind, concerns that survive past iteration 2 — none of it is aggregated across waves. Without instrumentation, no improvement signal. Now unblocked: Wave 1 shipped persona_versions (so metrics can be sliced by persona generation), and Wave 2 shipped reviewer_verdicts[] (so concern kinds and rejection rates are derivable for free). The aggregator is the missing piece.

Plan: a new any-bound skill /stx-wave-metrics that reads every docs/waves/*/wave-state.json in the project and emits docs/wave-metrics.html (single-file, same visual register as the existing verse templates). Metrics: per-persona suspicious-change rate, iteration histogram, Reviewer rejection rate, time-to-green per tier, top-3 worst tasks. Also emits a JSON sidecar so a CI step can fail the build if the suspicious rate exceeds a threshold. Schedulable via /loop 7d /stx-wave-metrics.

cost: 2–3 days · unblocked by Waves 1+2 · target: v1.10
04

No codebase index.

The Architect rediscovers existing patterns by grep every wave. Sourcegraph Amp's competitive edge is precisely the persistent index. A local symbol-graph cache (Phase A, cheap) or embedding index (Phase C, expensive) would let the Architect cite existing patterns without re-exploring.

Plan: defer until Wave 3 ships and its metrics show whether the Architect's pattern-citation accuracy actually plateaus low. If it does, a Phase A symbol-graph (tree-sitter or ast-grep over the consuming repo, written to .claude/index/symbols.json) is the cheapest move. Phase B adds a curated patterns.jsonl appended on every architecture decision. Phase C (embeddings) only if A+B aren't enough.

cost: ongoing · highest ceiling · target: post-Wave-3
The next move · Wave 3

Build the aggregator. The schema fields are all there — persona_versions, reviewer_verdicts[], suspicious[], escalations[], iterations_used. Ship /stx-wave-metrics as a new any-bound skill that walks docs/waves/*/wave-state.json and emits a single-file HTML report plus a JSON sidecar. Two to three days. Moves the score from 9 → ~9.5 and gives the first real signal for whether Wave 4 (codebase index) earns its cost.

What to measure first: per-tier Reviewer rejection rate (calibration check — if it never rejects, the prompt needs tightening; if it rejects ≥50%, it may be too aggressive); concerns that persist past iteration 2 (these are the real-world failure modes); time-to-green per tier (which tier slows the wave most).
