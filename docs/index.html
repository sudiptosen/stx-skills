01 Walkthrough
Features become Waves of tasks — specialist agents solve them in parallel, behind three approval gates.
Click any box to reveal its detail below. 7 steps · 3 user gates · zero hand-holding between gates.
Step 1 · Install

You're on main and the skills aren't here yet

A big change is inbound. If this harmless skill set isn't installed in your project, run one line. The installer only writes to .claude/skills/ — nothing else in your project moves.

npx github:sudiptosen/stx-skills

Re-running is the supported way to pick up updates. Your settings file (.claude/stx-skill-settings.json) is left alone on re-install.

Step 2 · On main

The launching pad

Start every wave on main. Skills that spawn a worktree refuse to run inside one — main is the launching pad because it's where the wave-starter skills live.

git checkout main

If you're in a stale worktree, cd back to main first. The skill halts with a clear message if you forget.

Step 3 · /stx-feature

One sentence is enough to seed the wave

Type the slash command, describe the feature in plain words. The skill spawns a worktree at .claude/worktrees/<name>/ and runs the Analyst. No production code is touched — just a conversation about what you're building.

/stx-feature Implement multi-delete feature for Admins on Dashboard

For a single reproducible bug, use /stx-fix instead. Full reference → /stx-feature in the catalog.

Step 4 · Approve

Three cheap moments to redirect

Analyst writes requirement-verse.html, you approve. Architect writes architecture-verse.html, you approve — scope freezes. QA writes failing tests in qa-verse.html, you approve — dry-run boundary reached.

Each gate is your chance to redirect before the expensive Dev wave starts. Catching a misread requirement at Gate 1 costs minutes; catching it after the Dev wave costs hours.

Compare with the bug-fix flow → §3 Features vs bug fixes.

Step 5 · Dev wave

One agent per tier, in parallel

db / service / api / ui — each tier has a specialized Dev agent prompt. Multiple Dev agents run in parallel where their scope_paths don't overlap. QA re-runs every test after each Dev cycle — a Dev calling "green" doesn't end the loop until QA confirms independently.

Iteration caps. Soft 3 per task → Architect re-engages. Hard 5 → wave halts with a handoff doc.

Step 6 · /stx-pr-merge

One chain, ten steps, no babysitting

Run from inside the worktree when all tests are green. It commits, pushes, opens the PR, builds twice (once in the worktree, once after merge), squash-merges, refreshes main. One approval sentence covers the whole chain.

/stx-pr-merge

Full reference → /stx-pr-merge in the catalog.

Step 7 · Cleanup

Worktree gone, main updated

The same skill removes the worktree, deletes the local branch, leaves main rebuilt and clean. The wave artifacts (requirement-verse, architecture-verse, qa-verse, result.html) ride along in the squash commit — so they're preserved in main's history.

Change by running /stx-settings. See §6 Settings.

If a wave halted instead of finishing, run /stx-report first to write a single HTML report into main's docs/ — that survives worktree removal.

More → §4 Worktree lifecycle.

The launching pad

git checkout main

Step 3 · /stx-fix

Repro first, then describe it

The skill interviews you for the bug — title, issues (numbered), repro URL/steps, expected behavior. It spawns a worktree at .claude/worktrees/<name>/ and renders a complete prompt from a template.

/stx-fix Timer offset wrong after DST

For a multi-task feature, use /stx-feature instead. Full reference → /stx-fix in the catalog.

Step 4 · Approve

The test IS the spec

Read the rendered prompt. It includes title, issues (numbered), repro, expected behavior, scope, out-of-scope. Approve, edit a field, or cancel — this is the only pre-Dev gate.

There's no architecture phase because for a bug, the failing test is the architecture. The Coder makes that test pass.

Step 5 · QA → Coder loop

Failing test first, smallest change to make it pass

QA writes the failing test in the right runner (Playwright for UI bugs, Vitest for logic). Coder reads it and makes the smallest change. QA re-runs independently. Green? done. Red? back to Coder.

Hard cap: 5 iterations. Coder MUST NOT edit the test file — touching it halts the loop and escalates.

Step 6 · /stx-pr-merge

One chain, ten steps, no babysitting

Run from inside the worktree once the test is green. It commits, pushes, opens the PR, builds twice (once in the worktree, once after merge), squash-merges, refreshes main. One approval sentence covers the whole chain.

/stx-pr-merge

Full reference → /stx-pr-merge in the catalog.

Step 7 · Cleanup

Worktree gone, main updated

The same skill removes the worktree, deletes the local branch, leaves main rebuilt and clean. The fix test rides along in the squash commit — so the bug stays caught if it ever returns.

Change by running /stx-settings. See §6 Settings.

The mental model

01

main never holds half-done code

Every in-flight change lives in a worktree under .claude/worktrees/<name>/. main only sees commits that have been merged. Discard a worktree and you lose that work — main is unaffected.

02

Skills know where they belong

Every skill is main-bound, worktree-bound, or any-bound. Run from the wrong checkout, the skill halts. No nested worktrees, no targeting the wrong tree.

Everything else follows:
  • /stx-feature and /stx-fix start on main because they spawn a worktree.
  • /stx-pr-merge and /stx-report run inside the worktree because that's where the work lives.
  • /stx-checkin, /stx-image, /stx-help, /stx-help-html are branch-agnostic — they don't care.
  • The settings file lives in main's checkout; each new worktree gets a copy at creation.
02 Three types of commands
Where you run a skill tells you what it does.
main-bound
You're on main and want to start something. The skill spawns a worktree.
worktree-bound
You're in a worktree doing feature work. The skill operates on it (commit, merge, report, clean).
any-bound
Runs anywhere. Utilities: commit, image audit, docs. Branch-agnostic.
main-bound main
  • /stx-featureinitiate a multi-task wave for big features
  • /stx-fixsingle-bug QA→Coder loop
  • npx stx-skillsinstall / refresh skills
  • /stx-skill-settingsview/edit settings (planned)
any-bound any
  • /stx-checkincommit + push with security scan
  • /stx-imageaudit / resize project images
  • /stx-helpterse text help
  • /stx-help-htmlthis walkthrough
worktree-bound worktree
  • /stx-pr-mergecommit→PR→build→merge→cleanup
  • /stx-reportHTML report of worktree changes
  • /stx-feature --resumeresume halted wave (planned)
Why types matter. A wave-starter run inside a worktree would nest worktrees inside worktrees. A cleanup skill run from main would target the wrong tree. The type tells the skill where it's allowed to run.

Create · work · report · discard.

F /stx-featureMulti-agent feature wave. Analyst → Architect → QA → Dev. Three gates, tier-specialized Devs, suspicious-change tracking. main-bound v1.6

Drives a multi-agent feature wave. Spawns a worktree, runs Analyst → Architect → QA in sequence (each behind a user gate), then dispatches tier-specialized Dev agents under QA control until all tests are green.

Use when
  • Feature decomposes into multiple tasks across tiers
  • You want failing tests written before any code
  • You want a written architecture decision, not freeform
Don't use when
  • It's a single bug → /stx-fix
  • Refactor with no behavior change (no failing test)
  • You're already inside a worktree (main-bound)

Single chained approval

One sentence covers every named step: "Commit these, push, open a PR, squash-merge, refresh main, clean up the worktree." Failures still halt; unexpected files still need a fresh approval.

06 Settings
One file, checked in by default.
Settings is checked in by default. Lives at .claude/stx-skill-settings.json. Installer does not gitignore it. Project-bound, team-shared if committed — NOT personalized per-user yet (see pending #1).

Setting spotlight · auto-clean

stx-pr-merge.auto-clean — default true. Change in .claude/stx-skill-settings.json to keep the worktree around after merge.

Settings pre-fill, they don't bypass. Skills still run their interview — settings change the default answer, not whether the question fires.

Worktrees see a snapshot

The settings file is copied into the worktree at creation (not symlinked). Edits to main's file don't propagate to a worktree already in flight — intentional, the wave runs with the settings it was started with.
